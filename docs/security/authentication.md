# Authentication

Authentication ensures that we know *who* is behind a connection or a message.

## Why this matters for users

Authentication is the first gate of trust in RCLDX.
It does not decide what you are allowed to do (that is authorization), but it confirms that your identity is valid before any action is processed.

## What users should know

RCLDX has two authentication paths:

1. **Web access (human login)**
2. **MQTT access (software and broker login)**

These paths are intentionally separated.

### Web access (OTP)

For web actions, users authenticate with:

- callsign
- email-based OTP challenge/verification

After successful OTP verification, a short-lived web session is issued.
This session is used only for web/API user operations (for example profile updates or token management).

### MQTT access (token)

For MQTT connections, software does **not** use OTP.
It uses a long-lived MQTT token generated by the user.

In simple terms:

- OTP = web login
- Token = MQTT login

### Individuals and clubs

Individuals and clubs authenticate with the same model.
There is no separate “special” authentication logic for clubs.

### Account state gate

Authentication is allowed only when account state is active.
If an account is suspended or banned, authentication is denied.

## Practical user flow

1. Register user identity (callsign + email).
2. Login on web UI/API with OTP.
3. Create MQTT token from authenticated web session.
4. Configure token in your client software.
5. Connect to MQTT through approved entry points.

## Security notes for users

- Never share your MQTT raw token.
- Rotate/revoke tokens if software or device is compromised.
- Keep software updated to current integration guidance.
- Treat OTP and token as private credentials.
